## Architecture (as implemented)

Three execution contexts connected by message-passing:

```
Background Service Worker          Offscreen Document              Kernel Worker
(controllerMessenger)              (CapTP proxy to kernel)         (kernel + vats)

  messenger.call(action, args) <── chrome.runtime.sendMessage ──── E(hostApiProxy).invoke()
                                        │
                              registerKernelServiceObject() ────→ registers 3 services
                              launchSubcluster() ───────────────→ vendor vat boots
                                        │
                              queueMessage(kref, method, args) → calls vat methods
```

### Key discoveries during implementation

1. **`registerKernelServiceObject` IS on the kernel facet** — but requires the linked `@metamask/kernel-browser-runtime` and `@metamask/ocap-kernel` packages to be rebuilt. A stale webpack cache can cause the method to appear missing.

2. **`bundleSpec` is the only supported vat config** — `sourceSpec` is not yet implemented in the kernel worker. Vats must be pre-bundled.

3. **Bundle URLs must be absolute `chrome-extension://` URLs** — relative paths fail with "Failed to fetch" because the vat supervisor's `fetch()` runs inside a vat iframe. Use `chrome.runtime.getURL()` to resolve the path.

4. **`queueMessage` returns CapData, not plain values** — results come back as `{ body: string, slots: string[] }`. To call methods on a returned exo, extract the kref from `slots[0]`.

5. **`getPresence()` returns a local stub, not a CapTP proxy** — it has methods like `getKref()` and `iface()` but cannot be used with `E()` for method calls. Use `queueMessage(kref, method, args)` instead.

6. **All objects crossing vat boundaries must be `harden()`'d** — the kernel rejects unfrozen objects with "Cannot pass non-frozen objects". Always `harden()` records and arrays before returning from vat methods.

7. **Compartment endowments use positional API** — `new Compartment(endowments)` works; `new Compartment({ globals: endowments })` does not make endowments available in the vat's SES environment.

8. **Baggage requires durable values** — live exo objects cannot be stored in baggage. Capability persistence needs a different approach (e.g., store only `sourceCode` and re-evaluate on restart). Persistence is disabled for MVP.

## Vendor Vat API (as implemented)

**Admin facet** (root object, returned by `buildRootObject`):
- `bootstrap(_vats, services)` — stores `hostApiProxy` and `llmService` refs in closure (no baggage)
- `getPublicFacet()` — returns the public facet exo
- `vendCapability(request)` — convenience that delegates to `publicFacet.vendCapability(request)`
- `getCapabilities()` — returns all in-memory capability records
- `revokeCapability(id)` — removes from in-memory registry

**Public facet**:
- `vendCapability(request)` — calls `E(llmService).prompt(request)`, evaluates `sourceCode` in `new Compartment({ E, makeDefaultExo, hostApiProxy })`, returns hardened `CapabilityRecord`

**CapabilityRecord** (hardened, returned from vend):
```typescript
{ id, name, description, methodNames, exo, sourceCode }
```
The `exo` field is the live capability object. When returned through `queueMessage`, it appears in `slots[]` as a kref.

## Security Notes (MVP)

- **hostApiProxy exposes ALL controller actions** via the root messenger — no allowlist. Acceptable for MVP.
- **LLM-generated code** runs in a SES Compartment with only `E`, `makeDefaultExo`, `hostApiProxy` as endowments. The broad attack surface is `hostApiProxy.invoke()` accepting any action string.
- **JSON serialization boundary**: All data crossing `chrome.runtime.sendMessage` must be JSON-serializable.
